## Phase 1: 블록별 문서 컨텍스트 분리 (핵심)

### 문제
현재 `prompt_processor.py`의 `UNIFIED_PROMPT_TEMPLATE`은 `{pdf_text}` 자리에 전체 문서를 그대로 넣는다. 블록이 "사업성 평가"든 "페르소나 분석"이든 동일한 전문을 받는다.

### 해결: 블록별 RAG 컨텍스트 조회

**파일: `rag_helper.py` 확장**

```python
def get_block_relevant_context(
    block: dict,          # blocks.json의 블록 객체
    rag_system: dict,     # 이미 구축된 RAG 시스템
    top_k: int = 8
) -> str:
    """
    블록의 description + steps + narrowing 키워드를 쿼리로 변환해
    문서에서 관련 청크만 추출한다.
    """
    # 블록에서 쿼리 생성
    query_parts = [
        block.get("description", ""),
        block.get("end_goal", ""),
        " ".join(block.get("steps", [])),
        " ".join(str(v) for v in block.get("narrowing", {}).values() if isinstance(v, str))
    ]
    query = "\n".join(filter(None, query_parts))

    # RAG 검색
    contexts = retrieve_relevant_contexts(
        query=query,
        document_chunks=rag_system["chunks"],
        document_embeddings=rag_system["embeddings"],
        top_k=top_k
    )

    return "\n\n---\n\n".join([chunk for chunk, _ in contexts])
```

**파일: `dspy_analyzer.py` — `analyze_with_rag()` 메서드 추가**

`EnhancedArchAnalyzer`에 블록별 컨텍스트를 받아 분석하는 메서드 추가. 기존 `analyze()` 메서드를 유지하면서 RAG 컨텍스트를 우선 사용하는 분기 추가.

**파일: `pages/3_Document_Analysis.py` — CoT 루프 수정**

블록 분석 시작 전 RAG 시스템을 한 번 구축(세션 상태 캐시)하고, 각 블록 실행 시 `get_block_relevant_context(block, rag_system)`으로 블록 전용 컨텍스트를 생성해 프롬프트에 주입.

```python
# 현재 (수정 전)
prompt = process_prompt(block, pdf_text=st.session_state.pdf_text)

# 수정 후
if rag_system:
    block_context = get_block_relevant_context(block, rag_system)
    prompt = process_prompt(block, pdf_text=block_context)
else:
    prompt = process_prompt(block, pdf_text=st.session_state.pdf_text)
```

RAG 시스템은 문서 업로드 후 최초 1회 구축하고 `st.session_state.doc_rag_system`에 캐싱.

## Phase 2: 하이브리드 검색 (키워드 + 의미) ✅ 완료

> `차이 분석.md` 제안 1: "도시 고유명사(지명, 법령명)와 정책 방향을 동시에 검색"

### 문제
현재 `rag_helper.py`는 임베딩 유사도(코사인)만 사용한다. "강남구", "도시재생 뉴딜", "용도지역 2종 일반주거" 같은 고유 키워드는 의미 검색으로 잘 안 잡힌다.

### 상세 설계

**스코어 결합 방식: Reciprocal Rank Fusion (RRF)**
- 단순 가중합 대신 RRF 사용: `score = 1 / (k + rank)` (k=60)
- 이유: BM25 점수와 코사인 유사도는 스케일이 달라 직접 합산 시 한쪽이 지배함. RRF는 순위(rank)만 사용하므로 스케일 무관하게 안정적.

**한국어 토크나이저: `_tokenize_korean()`**
- 공백·특수문자 기준 분리 → 어절 토큰
- 한국어 2자 이상 서브스트링(bigram) 추가 → "준주거지역"에서 "준주거", "주거지" 등 검색 가능
- 이유: 한국어 복합명사("도시재생뉴딜", "제2종일반주거지역")는 붙여쓰거나 띄어쓰기가 불일치하는 경우가 많아 bigram이 효과적

**BM25 인덱스 사전 구축**
- `build_rag_system_for_documents()` 반환 dict에 `bm25_index` 키 추가
- `rank_bm25` 미설치 시 `None` → 임베딩 단독으로 폴백

**`get_block_relevant_context()` 수정**
- RAG 시스템에 `bm25_index`가 있으면 `hybrid_retrieve()` 호출
- 없으면 기존 `retrieve_relevant_contexts()` 호출

### 변경 파일
- `requirements.txt`: `rank-bm25>=0.2.2` 추가
- `rag_helper.py`: `_tokenize_korean()`, `hybrid_retrieve()` 추가; `build_rag_system_for_documents()` · `get_block_relevant_context()` 수정

## Phase 3: 출처 매핑 및 환각 감지

> `차이 분석.md` 제안 2: "모든 분석 문장에 원본 데이터 매핑 + 환각 위험 점수"

### 상세 설계

**방식: LLM 호출 없는 경량 검증 (BM25 + Jaccard)**
- LLM을 추가로 호출하면 API 비용과 응답 시간이 크게 늘어남
- 대신 BM25로 주장 문장에 가장 근접한 원본 청크를 찾고, Jaccard 유사도로 신뢰도를 수치화
- 완전히 결정론적이고 추가 API 호출 없음

**핵심 주장 추출: `extract_key_claims(text, max_claims=6)`**
- 분석 결과를 문장 단위로 분리
- 필터 조건: 30자 이상 + (숫자 포함 OR 한글 고유 패턴 포함)
- 우선순위: 수치/법령명/지명이 있는 문장 → 정책/전략 문장
- 최대 `max_claims`개 반환

**신뢰도 계산: `verify_claim(claim, chunks, bm25_index)`**
1. BM25로 가장 유사한 청크 찾기 (랭킹)
2. claim ↔ 최우선 청크 간 Jaccard 유사도 계산 → 신뢰도
3. 반환: `{"claim", "evidence", "confidence", "is_grounded"}`

**신뢰도 기준**
- 0.15 이상 → 녹색 "근거 있음"
- 0.05~0.15 → 노란색 "부분 근거"
- 0.05 미만 → 빨간색 "근거 부족"

*(Jaccard는 임베딩 코사인과 달리 0.15가 실제로 꽤 높은 중복도임)*

**저장**: `st.session_state.cot_verifications[block_id]` — 재검증 없이 재표시 가능

**UI 위치**: 결과 탭 내 `st.expander("🔍 출처 검증")` — 기본 접힌 상태

### 변경 파일
- `rag_helper.py`: `_jaccard_similarity()`, `extract_key_claims()`, `verify_claim()`, `verify_analysis()` 추가
- `pages/3_Document_Analysis.py`: 결과 탭에 출처 검증 expander 추가, `cot_verifications` 세션 상태 추가

**저장 시점**: 파일 분석 완료("파일 분석 완료 확인") 버튼 클릭 시

**저장 내용**:
- 파일 바이너리 → Supabase Storage 버킷
- 파일 경로(URL) + 메타 → `analysis_sessions.session_data` 의 `file_storage_path`, `file_analysis` 키

**복원 시점**: 세션 로드 시 `file_storage_path`가 있으면 → Supabase Storage에서 다운로드 → 메모리에 복원

**구현 파일**: `auth/file_storage.py` (신규)

```python
def upload_project_file(user_id, project_id, filename, file_bytes) -> str:
    """Supabase Storage에 파일 업로드 후 path 반환"""

def download_project_file(file_path: str) -> bytes:
    """Supabase Storage에서 파일 바이너리 다운로드"""
```

**폴백**: 파일 크기 > 20MB 이면 "파일이 너무 커서 저장 불가, 다음 세션에 다시 업로드 필요" 안내

### Phase 7-C: 자동 저장 확장 (새 세션 상태 포함)

**`save_work_session()` 에 추가할 키**

```python
추가 저장 키:
- 'file_analysis'         # 파일 메타 (방법, quality_score, file_type 등)
- 'file_storage_path'     # Supabase Storage 경로
- 'site_fields'           # Mapping 필지 정보 (dict)
- 'downloaded_geo_data'   # 필지 polygon GeoJSON (dict) — 용량 주의, feature_count > 50 이면 스킵
- 'cot_verifications'     # 출처 검증 결과 (dict)
- 'urban_indicator_results' # 도시 지표 (dict)
- 'document_summary'      # 문서 요약
- 'block_spatial_data'    # 블록별 공간 레이어 연동 설정
```

**입력 필드 자동 저장 (debounce)**

현재: 파일 업로드 완료 버튼 클릭 시만 저장
변경 후: 아래 필드 변경 시 3초 debounce 후 자동 저장
- `project_name`, `location`, `project_goals`, `additional_info`
- 블록 선택 변경 시
- Mapping "분석에 활용" 클릭 시 (이미 site_fields가 session_state에 세팅됨)

구현: `auth/session_init.py`의 `auto_save_trigger()` 활용 (현재는 5초, 3초로 단축)

### 구현 순서 (Phase 7 내부)

| 순서 | 작업 | 변경 파일 | 이유 |
|------|------|-----------|------|
| 7-A | DB 스키마 (projects 테이블) | Supabase SQL | 7-B~D의 기반 |
| 7-C | 자동 저장 확장 (새 키) | `auth/session_init.py` | 가장 쉽고 즉각 효과 |
| 7-E | 저장 상태 표시 UI | `pages/3_Document_Analysis.py` | 사용자 불안감 해소 |
| 7-B | 파일 영속화 | `auth/file_storage.py` (신규), `pages/3_Document_Analysis.py` | 파일 재업로드 불편 해소 |
| 7-D | 프로젝트 선택 UI | `auth/project_manager.py` (신규), `pages/3_Document_Analysis.py` | 다중 프로젝트 필요 시 |

> **7-A → 7-C → 7-E 순으로 먼저 구현**하면 DB 스키마 변경 없이도 즉시 체감 개선 가능.
> 7-A는 7-B·7-D 구현 전에 완료되어야 한다.
